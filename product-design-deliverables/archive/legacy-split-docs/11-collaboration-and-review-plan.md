# 11 协作与评审机制

## 1. 文档目标

明确青搭在方案推进阶段的协作分工、评审节奏和输出物，避免只写文档不落地。

## 2. 参与角色

- 产品经理：负责需求定义、流程设计、优先级和验收标准
- 设计师：负责信息架构、交互方案和视觉表达
- 前端研发：负责用户端与后台端页面实现
- 后端研发：负责审核、消息、数据、权限等服务能力
- 测试：负责测试方案、核心链路验证和异常场景覆盖
- 运营：负责内容策略、活动策略和上线前运营准备

## 3. 协作方式

### 3.1 需求阶段

- 产品输出目标、流程和 PRD
- 设计参与结构与交互讨论
- 研发参与可行性评估与范围收敛

### 3.2 设计阶段

- 设计输出页面结构、主流程稿、关键交互稿
- 产品确认流程完整性与边界条件
- 研发提前识别实现风险

### 3.3 开发阶段

- 产品维护需求变更清单
- 研发按模块联调
- 测试提前介入核心场景用例

### 3.4 上线阶段

- 产品确认验收标准达成
- 运营准备首批内容与活动
- 数据同学或研发确认埋点完整

## 4. 评审节点建议

| 节点 | 评审内容 | 输出 |
| --- | --- | --- |
| 节点一 | 目标、用户、业务闭环 | 方案方向确认 |
| 节点二 | 核心流程、模块结构、优先级 | 范围确认 |
| 节点三 | PRD、原型、规则设计 | 开发前定稿 |
| 节点四 | 联调结果、验收标准、埋点 | 上线确认 |
| 节点五 | 数据复盘、问题清单 | 下一版规划 |

## 5. RACI 建议

| 事项 | 产品 | 设计 | 前端 | 后端 | 测试 | 运营 |
| --- | --- | --- | --- | --- | --- | --- |
| 目标与范围定义 | R | C | C | C | C | C |
| 流程设计 | R | C | C | C | C | C |
| 原型设计 | A | R | C | C | C | C |
| 技术方案 | C | C | R | R | C | I |
| 验收标准 | R | C | C | C | R | C |
| 上线准备 | A | I | C | C | C | R |

说明：

- R：主负责
- A：最终负责
- C：协作参与
- I：知会

## 6. 会议节奏建议

- 周初：版本目标对齐
- 周中：流程或设计评审
- 周末：开发进度、风险、阻塞同步
- 上线前：专项验收会
- 上线后：复盘会

## 7. 风险升级机制

- 涉及范围膨胀时，由产品经理发起收敛讨论。
- 涉及核心链路断裂时，优先处理首页、详情、发布、审核四大模块。
- 涉及数据不可追踪时，上线前必须补齐埋点，否则延后上线。
