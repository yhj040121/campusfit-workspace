# 06 核心业务流程设计

## 1. 文档目标

定义青搭的主流程、分支流程和异常流程，作为页面设计与后端逻辑的依据。

## 2. 主流程一：新用户冷启动流程

`启动 App -> 登录/注册 -> 完善基础信息 -> 进入首页 -> 浏览首批推荐内容 -> 触发首次互动或收藏`

设计重点：

- 首次进入不要让用户面对空白页面。
- 首屏内容需要突出校园场景和标签信息。
- 注册后尽快让用户完成一次有效互动。

异常处理：

- 未登录用户可浏览部分内容，但互动和收藏时引导登录。
- 资料未完善用户可先浏览，再在关键节点补录。

## 3. 主流程二：内容消费与导购流程

`首页/搜索 -> 内容卡片 -> 内容详情 -> 点赞/评论/收藏 -> 商品信息区 -> 跳转外部链接`

设计重点：

- 卡片到详情的转化要突出场景、风格、预算信息。
- 详情页要把内容价值和导购价值同时表达清楚。
- 跳转前可设置确认页，避免误触。

异常处理：

- 商品链接失效时提示“链接不可用”，并保留收藏内容。
- 外链跳转失败时提供复制链接或稍后重试。

## 4. 主流程三：搜索与标签筛选流程

`进入搜索页 -> 选择关键词/标签/预算/场景 -> 查看结果 -> 进入详情 -> 互动或跳转`

设计重点：

- 搜索不仅支持关键词，还要支持标签和场景组合。
- 搜索结果页要明确显示筛选条件，便于用户调整。

异常处理：

- 无结果时提供热门标签、相关推荐和发布建议。

## 5. 主流程四：创作者发布流程

`进入发布页 -> 上传图片/内容 -> 编辑文案 -> 选择场景标签 -> 绑定商品 -> 提交审核 -> 审核通过后发布`

设计重点：

- 发布流程字段不要过重，先确保核心信息完整。
- 标签是平台结构化能力的关键，必须做成必填或强引导项。
- 草稿箱要支持中途退出再编辑。

异常处理：

- 图片上传失败可重试。
- 商品链接校验失败时阻止提交并提示原因。
- 审核不通过时要返回明确驳回原因。

## 6. 主流程五：社交互动与消息回流流程

`用户点赞/评论/关注 -> 系统生成消息 -> 创作者收到通知 -> 创作者回访内容或用户主页 -> 二次互动`

设计重点：

- 消息中心要区分互动消息、系统消息、合作消息。
- 关注关系和收藏行为要能驱动回访。

异常处理：

- 被举报或下架的内容不再触发新增互动通知。

## 7. 主流程六：后台审核治理流程

`内容待审 -> 审核人员查看 -> 通过/驳回/下架 -> 结果通知创作者 -> 数据记录`

设计重点：

- 审核动作要标准化，避免同类问题处理口径不一致。
- 下架、封禁、申诉、恢复要有状态流转记录。

异常处理：

- 高风险内容支持升级给管理员复审。
- 多次违规用户可进入重点监控名单。

## 8. 主流程七：举报与申诉流程

`用户举报 -> 系统记录 -> 审核复核 -> 采取处理措施 -> 通知相关方 -> 支持申诉`

设计重点：

- 举报原因要结构化，便于统计和追踪。
- 处罚不是终点，必须有申诉入口和处理闭环。

## 9. 流程设计原则

- 每条主流程都要能落到页面和状态上。
- 每条核心流程都要有异常路径和回退路径。
- 每条流程都要可埋点、可监控、可复盘。
