# 青搭毕业设计说明书第 4 章初稿

说明：本文件承接 `07-graduation-design-draft-ch3-framework.md`，对应毕业设计说明书第 4 章“系统开发与实现”。内容依据当前工作区中的 `web/`、`mobile-uniapp/`、`admin-web/`、`backend-java/` 与数据库脚本整理而成，写法偏向论文正文，可在后续补充截图、接口时序图和测试结果后直接并入说明书。

## 第 4 章 系统开发与实现

### 4.1 系统开发关键技术

#### 4.1.1 Vue 3 与 Vite 用户端技术

青搭的 Web 用户端基于 Vue 3 与 Vite 构建。Vue 3 提供了组件化开发能力，使首页内容流、详情页、发布页和个人中心等页面可以按业务边界拆分为独立组件；Vite 则提供了更轻量的本地开发体验与更快的热更新能力。当前 Web 工程通过 `web/src/main.js` 完成应用挂载，通过 `web/src/router/index.js` 完成用户端路由组织，并通过 `web/src/api/client.js` 对后端接口进行统一封装。

在状态管理方面，项目没有引入额外的状态库，而是使用基于 Vue 响应式能力实现的 `appStore` 作为全局状态中心。该方案集中维护公告、活动、帖子、草稿、个人资料、激励中心和消息等数据，并通过 `loadPublicData`、`loadPrivateData` 与 `refreshAll` 等方法完成公共数据和登录后私有数据的统一加载。这种实现方式结构清晰，比较适合毕业设计阶段的中小型平台项目。

#### 4.1.2 uni-app 移动端技术

移动端采用 uni-app 技术方案实现。该方案能够在统一代码结构下兼顾 H5 和移动端运行场景，降低多端开发与维护成本。当前工程按照 uni-app 的标准目录组织页面与公共模块，页面位于 `mobile-uniapp/pages/`，网络请求与接口封装位于 `mobile-uniapp/common/`。

其中，`common/request.js` 负责基础网络请求封装，自动拼接后端地址并携带登录态 Token；`common/api.js` 则在此基础上进一步封装用户登录、帖子详情、发布、草稿、活动、消息等业务接口，同时为图片上传提供了独立的 `uni.uploadFile` 封装。通过这种方式，移动端页面层可以聚焦界面与交互，而将网络与鉴权逻辑下沉到公共层处理。

#### 4.1.3 Spring Boot 3 与 JdbcTemplate 后端技术

后端采用 Spring Boot 3 构建，JDK 版本为 21。与传统分层应用类似，系统按照 controller、service、vo、dto 等结构组织模块，接口统一挂载在 `/api` 路径下，后台接口统一挂载在 `/api/admin` 路径下。

与部分使用 ORM 框架的项目不同，青搭后端当前主要基于 `JdbcTemplate` 实现数据库访问。该方案对于毕业设计项目而言具有两个优点：一是 SQL 结构清晰，便于将数据库设计与具体业务逻辑一一对应；二是减少了额外框架配置，使开发过程更可控。当前后端已实现认证、内容、草稿、活动、消息、个人中心、合作、上传和后台治理等模块，覆盖了毕业设计所需的完整业务主线。

#### 4.1.4 MySQL 数据库技术

系统数据库采用 MySQL 8。数据库结构围绕用户、内容、互动、活动、激励、治理等核心对象进行设计，建表脚本集中在 `backend-java/src/main/resources/schema.sql` 中。数据库不只承担用户端内容展示的数据存储功能，还支撑后台审核、商家合作、激励结算与提现审核等平台侧能力。

相较于单纯的演示原型，青搭的数据库结构已经形成了较完整的平台型系统雏形。例如，`post` 表与 `post_image`、`post_tag`、`product_link` 表共同支撑内容与导购功能，`creator_cooperation`、`commission_record`、`creator_withdraw_request` 表支撑创作者激励链路，`message_notification` 和 `official_announcement` 表支撑平台通知与公告能力。

#### 4.1.5 接口通信与统一鉴权机制

用户端 Web 通过 Axios 请求后端接口，移动端则通过 `uni.request` 与 `uni.uploadFile` 请求后端接口。两端都采用 Bearer Token 形式传递用户登录态。后端通过 `UserAuthInterceptor` 对 `/api/` 路径请求进行统一拦截，解析 Token 后将用户会话写入请求上下文，并通过 `UserAuthContext` 向业务模块传递当前用户信息。

后台管理端使用独立的登录态校验机制。`AdminAuthInterceptor` 会对 `/api/admin` 路径进行拦截，未登录请求统一返回 401 状态。前端后台页面又在路由层增加了角色判断逻辑，从而形成“后端鉴权 + 前端菜单与路由控制”双层保护机制。

#### 4.1.6 阿里云 OSS 图片上传能力

青搭在图片资源管理上引入了阿里云 OSS。后端 `UploadServiceImpl` 通过 AWS S3 兼容 SDK 访问 OSS，对图片类型、大小和扩展名进行校验后再上传到对象存储，并根据配置生成可公开访问的图片地址。系统当前支持帖子图片、头像和个人主页封面上传，上传接口分别对应 `/api/uploads/images`、`/api/uploads/avatar` 和 `/api/uploads/profile-cover` 等路径。

这一方案既满足了毕业设计展示中“真实上传并回填 URL”的要求，也为后续移动端和 Web 用户端统一加载远程图片资源提供了基础。

### 4.2 开发环境与工程结构

#### 4.2.1 开发环境

根据当前工程配置，系统开发环境主要包括：

- 操作系统：Windows 开发环境
- JDK：21
- 后端框架：Spring Boot 3.3.5
- 数据库：MySQL 8
- 前端：Vue 3 + Vite
- 管理后台组件库：Element Plus
- 移动端：uni-app
- 构建工具：npm、Maven
- 移动端开发工具：HBuilder X

后端核心配置定义在 `backend-java/src/main/resources/application.yml` 中，数据库连接、上传限制、JWT 密钥和 OSS 参数均可通过环境变量覆盖。用户端与后台端分别通过 `.env.backend-local` 和 `.env.backend-server` 等文件切换本地联调与服务器联调环境。

#### 4.2.2 工程结构

当前工作区采用多模块结构组织，各模块职责明确：

- `web/`
  - 用户端 Web
- `mobile-uniapp/`
  - 用户端移动端
- `admin-web/`
  - 管理后台
- `backend-java/`
  - 后端服务
- `database/`
  - 数据库相关资源
- `product-design-deliverables/`
  - 产品、论文与答辩文档

这种结构既符合前后端分离项目的开发方式，也便于在毕业设计说明书中分别描述“用户端实现”“后台实现”和“后端实现”。

### 4.3 后端模块实现

#### 4.3.1 用户认证与会话模块实现

用户认证模块主要由 `UserAuthController` 与 `UserAuthServiceImpl` 实现。接口层提供验证码发送、登录、注册、获取当前用户信息和退出登录等接口。业务层支持验证码登录和密码登录两种方式，其中验证码流程在当前演示环境下统一使用固定验证码 `040121`，便于毕业设计演示和联调测试。

注册成功后，系统会同时完成三项操作：向 `app_user` 表插入账号信息、向 `user_profile` 表初始化资料信息、向 `message_notification` 表插入欢迎通知。登录成功后，系统会通过 `UserJwtTokenService` 生成用户 Token，并在后续请求中由拦截器解析。该设计说明系统并非仅完成登录页面展示，而是已经实现了完整的账号创建与登录态维护能力。

#### 4.3.2 内容、互动与导购模块实现

内容模块由 `PostController` 与 `PostServiceImpl` 共同实现，主要覆盖以下能力：

- 推荐内容流获取
- 我的发布、点赞、收藏列表获取
- 条件搜索
- 内容详情查询
- 评论创建与删除
- 点赞与收藏切换
- 商品导购信息读取与点击追踪

在内容发布方面，`PostServiceImpl.create` 会先对标题、描述、标签和图片进行规范化处理，再向 `post` 表写入主记录，并同步完成 `post_image`、`post_tag`、`product_link`、活动绑定和合作绑定等关联数据写入。系统默认新发布内容的 `audit_status` 为待审核状态，这保证了平台内容在公开展示前经过治理流程。

在互动方面，点赞、收藏和评论都采用事务化处理方式。例如，点赞操作不仅会在 `post_like` 表中插入或删除关系记录，还会同步维护 `post.like_count` 计数，并在必要时向作者写入互动通知。评论功能会写入 `post_comment` 表，并同步更新帖子评论数。商品导购方面，系统会在用户跳转外部商品前先通过 `product_link_click` 记录点击行为，为后续激励与效果分析提供数据基础。

#### 4.3.3 草稿与重新编辑模块实现

草稿模块由 `DraftServiceImpl` 实现。系统允许用户在内容尚未准备完成时先将发布数据保存为草稿，草稿内容包括标题、描述、图片 URL 列表、标签列表、商品链接、价格、活动编号和合作单编号等信息。图片与标签信息以 JSON 形式存储在 `post_draft` 表中。

当用户选择“发布草稿”时，系统会先校验草稿是否达到可发布条件，再将草稿内容转换为正式的 `PostCreateRequest`，并调用 `postService.create` 完成正式发布。发布成功后，草稿会被软删除。通过这种实现方式，草稿并不是孤立的临时存档，而是与正式发布链路高度复用，提升了系统整体一致性。

#### 4.3.4 个人中心与创作者激励模块实现

个人中心模块由 `ProfileServiceImpl` 实现，主要包括资料展示、资料编辑、关注关系维护、激励中心和提现申请等能力。系统通过聚合 SQL 统计用户关注数、粉丝数、获赞数、合作数以及激励金额，并将这些数据统一返回给个人中心页面展示。

在创作者激励方面，系统会从 `commission_record` 表中统计累计激励、待结算金额、可提现金额和已提现金额，同时读取最近的结算记录与提现记录。提现申请时，系统会根据最小提现门槛和手续费率生成申请记录，并写入 `creator_withdraw_request` 表。这部分实现使青搭在论文层面不仅是一个“内容展示系统”，也是一个具备平台经营能力的内容产品。

#### 4.3.5 后台治理模块实现

后台治理能力主要由 `AdminDashboardServiceImpl` 提供。当前已实现的数据看板、用户管理、内容审核、公告管理、活动管理、合作管理、商家管理、结算记录与提现申请审核等功能，均通过该服务或其关联控制器向后台前端提供数据。

以内容审核为例，后台首先通过 `listContentAuditItems` 获取审核列表，再通过 `getContentAuditDetail` 获取单条内容详情，包括图文、作者、互动数据、商品链路、合作信息与审核状态。审核人员执行“通过”或“驳回”后，系统会修改内容审核状态，从而影响前台是否可见。通过这一设计，平台形成了完整的“用户投稿 -> 后台审核 -> 前台展示”治理闭环。

#### 4.3.6 图片上传模块实现

图片上传模块由 `UploadServiceImpl` 实现。系统会校验上传文件是否为空、是否超过 10MB、是否属于 jpg、jpeg、png 或 webp 格式，验证通过后再写入 OSS。对象路径会根据上传类型和当前用户生成不同目录前缀，例如帖子图片、头像和封面图分别存放在不同路径下。

该模块的实现说明系统在资源管理上并未停留在本地假数据层面，而是已经具备真实上传和 URL 回填能力，这对毕业设计展示具有较高说服力。

### 4.4 前端多端实现

#### 4.4.1 Web 用户端实现

Web 用户端以路由驱动方式组织页面。`web/src/router/index.js` 中定义了首页、公告、活动、消息、合作、详情、发布、个人中心、激励中心和资料编辑等页面路由。`web/src/api/client.js` 将后端接口按业务进行封装，页面无需直接拼接 URL，只需调用语义化接口函数。

全局状态层 `appStore` 将公告、活动、帖子、草稿、个人资料、激励数据等内容统一标准化后存入响应式状态中。公共数据通过 `loadPublicData` 批量拉取，私有数据通过 `loadPrivateData` 批量拉取，从而实现“首屏数据”和“登录后数据”分层加载。该结构使前端在实现上更接近真实产品工程，而不是若干独立页面的简单拼接。

在页面实现层面，首页页面负责组织公告入口、活动入口和内容流展示，详情页负责图文轮播、点赞、评论、收藏、关注和导购跳转，发布页负责图片上传、活动选择、合作单绑定、草稿保存和正式提交。页面之间通过路由参数与全局状态联动，形成统一的用户端体验。

#### 4.4.2 uni-app 移动端实现

移动端工程采用页面目录加公共模块的方式组织。`common/request.js` 实现统一请求与错误处理，`common/api.js` 进一步定义业务接口。页面层则通过 `pages/index/`、`pages/detail/`、`pages/publish/`、`pages/profile/`、`pages/messages/` 等目录实现具体页面。

以发布页为例，移动端不仅支持图片上传、标题与描述填写、标签和预算选择，还支持活动预览、合作单绑定、草稿恢复与编辑模式切换。页面通过计算属性判断当前是否满足发布条件，并通过状态提示卡将“上传中”“保存中”“待完善”等状态实时反馈给用户。这种实现方式更贴合移动端内容生产场景，也较好体现了用户体验导向设计。

详情页则实现了图片轮播、作者资料、关注按钮、标签展示、关联活动入口、商品卡片、评论区与回复等功能，与 Web 用户端形成了业务层一致、表现层差异化的双端结构。

#### 4.4.3 管理后台实现

管理后台通过 `admin-web/src/router/index.js` 组织后台登录、数据看板、用户管理、内容审核、公告管理、活动管理、合作管理、商家管理、结算记录和提现申请等路由。后台路由不仅依赖登录态，还通过角色信息限制访问范围，从而使不同后台身份进入各自职责页面。

在接口层，`admin-web/src/api/admin.js` 将后台操作封装为登录、拉取看板、审核内容、管理公告、管理活动、管理商家、处理结算和提现等函数。页面层基于 Element Plus 的表格、弹窗、输入框和按钮等组件快速完成运营后台交互。

以内容审核页为例，系统先显示审核队列卡片，再通过弹窗形式展示单条内容详情。审核人员在同一页面即可查看图片、文案、互动数据和商品链路，并执行通过或驳回操作。该实现相较于简单表格式后台更有产品化特征，也更适合在毕业答辩中展示“后台治理系统”的完整性。

### 4.5 关键功能实现

#### 4.5.1 登录注册与资料初始化功能实现

系统登录注册功能通过前后端联动完成。前端提交手机号、验证码或密码后，后端认证模块负责校验账号并生成用户 Token。新用户注册成功后，系统会自动初始化用户资料表和欢迎消息记录，从而避免出现“注册成功但个人中心无数据”的问题。

这一功能的实现重点在于前后端统一使用手机号作为账号主标识，并通过拦截器在后续接口中自动识别当前用户。这样既简化了业务逻辑，也提高了系统整体一致性。

#### 4.5.2 首页内容流与详情互动功能实现

首页内容流功能通过后端推荐接口和前端状态层共同完成。前端一次性拉取公告、活动、推荐内容和标签选项，并在页面中组织为首页内容流。用户点击任意内容卡片后进入详情页，在详情页中可进行点赞、评论、收藏、关注作者和查看商品等操作。

互动行为并不是单纯修改前端按钮状态，而是会同步写入数据库并维护统计字段。例如，点赞会影响 `post_like` 关系表与 `post.like_count` 计数，评论会写入评论表并同步增加评论数，关注行为则会修改 `user_follow` 关系。通过这种方式，详情页真正成为内容消费与社区互动的核心节点。

#### 4.5.3 内容发布、草稿与重新编辑功能实现

发布功能是青搭系统的核心实现之一。用户在发布页上传图片、填写标题与描述、选择标签、补充商品链接和价格，并可选绑定活动或合作单。提交时，前端会先检查图片上传状态和表单完整性，再向后端提交正式发布请求。

若内容暂未准备完成，用户可以先保存为草稿。草稿会保存在后端数据库中，后续重新进入发布页时可恢复草稿继续编辑。对于审核未通过或需要修改的内容，系统也支持进入编辑模式重新提交审核。这一设计使内容生产链路从“一次性提交”提升为“可保存、可恢复、可重提”的完整创作流程。

#### 4.5.4 商品导购跳转与数据记录功能实现

青搭并未直接实现完整交易链路，而是采用“内容 + 外部导购”的轻导购模式。用户在详情页点击商品卡片后，会先读取商品跳转信息，再执行导购点击追踪，最后跳转外部平台。后端通过 `product_link_click` 表记录点击来源、目标地址、用户 ID 和时间等信息。

该实现与开题报告中“看穿搭 -> 查单品 -> 跳转购买”的目标相吻合，也为后续活动统计、商家合作评估和创作者激励提供了可量化的数据支撑。

#### 4.5.5 后台审核与运营管理功能实现

后台审核功能是平台治理的核心。当前实现中，所有新发布或修改后的内容都会先进入待审核状态，后台审核人员通过审核页进入弹窗查看详情，并执行通过或驳回操作。审核结果会影响内容能否进入公开内容流。

除了审核外，后台还实现了公告发布、活动配置、商家管理、合作管理、结算记录与提现审核等能力。这表明青搭并不是单一的前台展示系统，而是具有较完整后台运营支撑能力的平台型应用。

### 4.6 系统联调与运行说明

当前工程已经具备本地联调所需的基础结构。一般情况下，可按以下顺序启动系统：

1. 准备 MySQL，并执行建表与初始化脚本
2. 启动 `backend-java/` 后端服务
3. 启动 `admin-web/` 管理后台
4. 启动 `web/` 用户端 Web
5. 使用 HBuilder X 打开 `mobile-uniapp/` 并运行移动端

用户端 Web 和管理后台均支持本地后端联调与服务器后端联调两种模式，移动端通过公共配置确定当前请求基地址，后端则通过环境变量控制数据库、JWT 和 OSS 配置。该结构能够满足开发调试、演示答辩和后续扩展三类需求。
