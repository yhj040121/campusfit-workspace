# CampusFit 产品流程、架构与规则整合文档

## 1. 文档定位

本文档整合 CampusFit 的业务闭环、用户流程、信息架构、功能架构和关键规则，作为设计、研发和后台实现的主要依据。

## 2. 产品核心闭环

CampusFit 的核心闭环建议定义为：

`活动/推荐触达 -> 用户浏览内容 -> 互动反馈 -> 内容热度积累 -> 激励结算 -> 创作者持续发布 -> 平台与商家持续投入`

相比交易平台，这一闭环更强调：

- 内容增长
- 社区活跃
- 活动运营
- 激励驱动

## 3. 核心业务流程

### 3.1 新用户冷启动流程

`启动 App -> 注册/登录 -> 完善资料 -> 浏览推荐内容 -> 首次互动或关注 -> 接触活动入口`

目标：

- 让新用户尽快理解平台内容形态
- 引导用户从浏览走向互动

### 3.2 内容消费流程

`首页/搜索/专题 -> 内容卡片 -> 详情页 -> 点赞/评论/收藏 -> 关注作者或参与活动`

目标：

- 提升内容消费深度
- 提高内容互动率

### 3.3 内容发布流程

`进入发布页 -> 上传图片 -> 编辑文案 -> 选择标签/专题 -> 提交审核 -> 审核通过 -> 内容上线`

目标：

- 降低创作门槛
- 保证内容可审核、可管理、可参与奖励

### 3.4 活动参与流程

`查看活动说明 -> 发布活动内容 -> 通过审核 -> 进入活动榜单或统计池 -> 周期结算`

目标：

- 把活动和内容发布绑定
- 为平台增长和激励提供统一入口

### 3.5 审核治理流程

`内容待审 -> 审核通过/驳回 -> 上线或退回修改 -> 异常举报 -> 复核处理`

目标：

- 保障内容质量
- 保证活动和激励规则公平

### 3.6 激励结算流程

`统计互动数据 -> 清洗异常数据 -> 计算结算分 -> 审核复核 -> 发放奖励 -> 记录日志`

目标：

- 让激励规则可执行
- 让结算过程可追溯

## 4. 用户端信息架构

### 4.1 一级导航建议

- 首页
- 发布
- 活动
- 我的

如果想保留搜索独立入口，可作为首页顶部或活动页顶部能力，不一定必须独立成底部一级导航。

### 4.2 首页结构建议

- 顶部搜索
- 热门活动入口
- 推荐内容流
- 标签或专题快捷入口

### 4.3 我的页面结构建议

- 我的发布
- 我的收藏
- 我的活动
- 消息通知
- 资料设置

## 5. 后台信息架构

建议按“运营、审核、商家、结算、数据”五个方向设计：

- 控制台
- 内容审核
- 活动管理
- 创作者管理
- 商家管理
- 奖励结算
- 举报治理
- 数据看板
- 系统设置

## 6. 功能架构整合

### 6.1 用户端功能模块

- 账号体系
- 内容浏览
- 内容互动
- 搜索与标签
- 内容发布
- 活动参与
- 消息通知
- 个人中心

### 6.2 平台端功能模块

- 内容审核
- 专题活动配置
- 用户与创作者管理
- 商家合作管理
- 结算与奖励管理
- 举报与违规治理
- 数据分析

## 7. 核心规则体系

### 7.1 账号规则

- 未登录用户可浏览部分内容
- 登录后可进行互动、关注、收藏和发布
- 参与奖励活动的账号需满足基础资料要求

### 7.2 内容规则

- 内容需包含图片和基础文案
- 场景标签建议为必填
- 活动内容需绑定活动标签或专题
- 审核不通过内容不可公开展示

### 7.3 活动规则

- 每个活动有明确时间、主题和参与条件
- 活动内容需通过审核后方可进入统计
- 活动期间禁止重复投稿刷奖励

### 7.4 激励规则

- 激励以结算分为依据
- 异常数据需剔除或降权
- 违规内容取消奖励资格

### 7.5 举报与申诉规则

- 用户可举报搬运、低俗、刷量、恶意互动等问题
- 举报进入后台待处理队列
- 被处理方可申请复核

## 8. 数据与埋点建议

为保证后续活动、激励和商家合作能复盘，建议至少埋点以下行为：

- 首页曝光
- 内容详情点击
- 点赞、评论、收藏
- 关注行为
- 活动报名或参与
- 发布提交
- 审核通过/驳回
- 奖励发放

## 9. 异常场景处理

### 9.1 内容审核不通过

- 明确驳回原因
- 支持修改后重新提交

### 9.2 活动内容违规

- 取消活动资格
- 保留平台处罚记录

### 9.3 互动数据异常

- 暂缓结算
- 触发人工复核

### 9.4 商家预算不足或活动暂停

- 不影响普通内容展示
- 停止对应活动奖金池扩充

## 10. 设计结论

CampusFit 的产品结构应围绕“内容、活动、审核、激励”四大核心模块展开。只要这四部分结构清晰、状态闭环完整，前后台就能形成一套足够真实、足够专业的产品方案。
