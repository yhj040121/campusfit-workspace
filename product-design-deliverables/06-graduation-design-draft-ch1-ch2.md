## 第 1 章 绪论

### 1.1 研究背景与意义

随着移动互联网、社交媒体与电子商务的持续发展，大学生获取穿搭信息、完成消费决策和表达个体风格的方式正在不断变化。与传统以商品陈列和交易为中心的平台不同，当前年轻用户更加依赖真实内容、场景化表达与同龄群体经验来完成日常决策。尤其在高校场景中，穿搭并不仅仅是消费行为，还与社交表达、审美认同和自我展示密切相关。学生群体在面对“今天应该怎么穿”“某种场景下如何兼顾预算与实用性”等问题时，往往更倾向于参考贴近自身生活环境的内容，而不是单纯浏览商业化程度较高的广告或商品详情。

然而，从现有平台形态来看，大学生穿搭需求并没有被充分满足。一方面，综合类内容平台拥有大量时尚内容，但内容过于泛化，难以突出图书馆、自习、早八、社团活动、校园通勤等高频校园场景；另一方面，传统电商平台虽然具备较强的商品聚合和交易能力，却较少提供“如何搭配”“为什么这样搭配”“这种搭配是否适合校园日常”等具有参考价值的内容支持。对于预算有限、注重性价比和复用率的大学生而言，现有平台往往难以同时满足真实场景、低决策成本和内容可信度这三项需求。

基于上述背景，本文设计并实现了一套面向大学生群体的校园穿搭内容平台青搭。该平台以校园真实场景为基础，以内容发布、内容浏览、互动传播、活动激励和后台治理为核心结构，力图通过“内容驱动”的方式降低用户穿搭决策成本，提升内容消费效率，并为校园创作者提供稳定的内容表达和激励机制。与以交易为中心的平台不同，青搭更强调围绕内容形成持续增长的产品闭环，即通过优质内容吸引用户，通过互动与活动增强参与感，再通过激励与平台治理维持创作者供给和内容质量。

本课题的研究与实践具有一定的现实意义。首先，从用户价值层面看，平台聚焦大学生日常高频生活场景，能够提供更真实、更可模仿、更具预算友好性的穿搭参考，有助于解决校园穿搭内容分散、信息不对称和决策效率低的问题。其次，从平台设计层面看，本项目不仅关注用户端体验，还将后台审核、活动配置、商家合作、结算与提现等平台经营能力一并纳入设计范围，使系统具备较完整的业务闭环和落地逻辑。最后，从工程实践层面看，青搭采用前后端分离的系统架构，结合 Vue 3、Spring Boot、MySQL 与 uni-app 等技术实现用户端、管理后台、后端服务与数据库的协同运行，具有较强的综合设计与实现价值。

因此，围绕校园穿搭场景设计一套兼顾内容消费、内容创作、平台治理与轻商业化能力的系统，不仅具有一定的产品创新意义，也能够体现移动互联网产品从需求分析到架构设计、从功能实现到系统联调的完整实践过程。

### 1.2 相关研究与行业现状

#### 1.2.1 校园垂直内容平台的发展现状

近年来，内容社区型产品在移动互联网领域持续发展，垂直内容平台也逐渐由泛兴趣分发向细分场景运营演变。对于校园用户而言，其内容需求具有较强的场景性和同龄群体特征，用户通常更加关注真实、可借鉴和与日常生活紧密相关的内容形式。在这一背景下，围绕校园生活、学习经验、社交表达和兴趣分享的内容产品逐渐增多，但真正聚焦校园穿搭这一细分方向的成熟平台相对有限。

现有校园类产品多集中于校园社交、校园服务、二手交易或学习交流等方向，穿搭内容往往只是附属表达的一部分，缺乏独立的信息架构和持续运营机制。与此同时，部分时尚社区虽然积累了丰富的图文内容，但其内容语境更偏向都市生活、品牌种草或商业化展示，与大学生日常校园场景存在一定差异。因此，从校园垂直内容平台的发展现状来看，面向高校用户构建一个以真实场景和低决策成本为核心特征的穿搭内容平台，仍具有较明确的需求空间。

#### 1.2.2 穿搭内容平台与导购模式的发展现状

穿搭类平台的发展经历了从“图片展示”向“内容种草”再到“内容导购”的演进过程。传统导购平台更重视商品链接、品牌推荐和购买转化，而近年的内容平台则更强调搭配思路、场景表达和用户参与感。对于年轻用户而言，单纯的商品列表已经难以满足其决策需求，内容平台通过穿搭图文、短视频、互动评论和专题活动等方式，使用户在消费之前先完成信息理解与风格判断。

不过，在导购模式方面，现有平台普遍存在两类问题。第一，部分平台过度依赖商业推广，内容参考价值被商品营销弱化，用户难以判断内容是否真正适合自身场景。第二，许多平台将重点放在成交闭环，而忽视了校园用户更在意的预算控制、单品复用和场景适配等因素。对于青搭这类校园穿搭平台而言，更合适的方式并非直接复制完整交易平台模型，而是以内容参考和轻导购为主，在保留商品跳转和商家合作能力的同时，将平台核心价值建立在内容质量、活动参与和创作者激励之上。

#### 1.2.3 前后端分离开发模式的发展现状

从系统实现角度看，前后端分离已经成为当前 Web 与移动应用开发中的常见模式。该模式将界面层、业务接口层和数据存储层相互解耦，使得用户端、管理后台和后端服务能够在统一的数据接口下并行开发。对于功能结构较复杂、同时包含用户端和运营后台的平台型产品而言，前后端分离模式有利于提高开发效率、降低耦合度，并增强系统后续扩展能力。

在前端技术方面，基于组件化思想的 Vue 生态能够较好支持内容流页面、详情页、表单页和后台管理页等多样化场景；在后端技术方面，Spring Boot 具备配置简洁、开发效率较高和生态成熟等特点，适合中小型平台项目快速搭建接口服务；在移动端方案上，uni-app 通过统一语法支持多端适配，可以降低移动应用原型落地和后续扩展成本；在数据存储层面，MySQL 作为成熟的关系型数据库，能够较好支撑用户、内容、活动、互动、消息、结算等结构化业务数据的组织与管理。

综上所述，从行业应用形态和技术实现路径来看，采用前后端分离架构设计一个聚焦校园场景的穿搭内容平台，既符合当前产品发展趋势，也具备明确的实现基础。

### 1.3 可行性分析

#### 1.3.1 技术可行性分析青搭在技术方案上采用了较为成熟且易于落地的组合方式。用户端 Web 项目基于 Vue 3、Vite、Vue Router 与 Axios 实现，能够满足内容流展示、详情交互、标签筛选、发布表单、消息通知等典型前端需求；管理后台项目基于 Vue 3 与 Element Plus 实现，适合快速构建数据看板、表格筛选、审核操作、活动配置和商家管理等后台功能；后端服务基于 Spring Boot 3 与 JDBC 进行开发，能够较好承接认证、内容管理、活动管理、消息通知、上传服务和后台治理等业务逻辑；数据库采用 MySQL 8，用于存储用户、帖子、标签、活动、评论、收藏、消息、商家、结算等核心结构化数据；移动端则基于 uni-app 工程进行实现，便于后续扩展到更贴近真实移动使用场景的端侧体验。

从项目现有工程情况看，本系统已具备较完整的技术落地基础。工作区中已经包含 `web/`、`admin-web/`、`backend-java/`、`mobile-uniapp/` 和 `database/` 等独立目录，说明系统在架构上已完成用户端、后台端、后端服务与数据库的拆分设计。后端已实现用户认证、帖子、标签、活动、消息、个人中心、后台管理等模块，数据库中也已设计用户表、内容表、活动表、草稿表、消息表、商家表、提现申请表和管理员表等核心数据结构。因此，从开发语言、框架成熟度和现有工程基础来看，本项目具备良好的技术可行性。

#### 1.3.2 业务可行性分析青搭的业务定位较为清晰，即围绕大学生的校园穿搭决策需求，建立一个以内容为核心的轻社区平台。该定位回避了直接进入重交易、重供应链和重履约体系的高复杂度路径，而是优先聚焦真实场景、内容供给、用户互动和平台激励，降低了首版产品的业务风险。对于高校用户而言，穿搭需求高频、表达意愿较强、内容传播自然具备社交属性，因此平台具备内容增长和互动扩散的业务基础。

同时，青搭并未停留在单纯前台展示层面，而是将后台审核、活动管理、公告管理、商家管理、结算记录与提现审核等能力纳入系统结构。这种设计方式使平台能够从一开始就考虑内容质量控制、活动运营承接和轻商业化支撑，从而避免“有前台、无治理、无闭环”的设计缺陷。项目已形成的业务闭环可概括为：用户浏览内容并产生互动，创作者发布内容并参与活动，平台通过审核、活动和消息机制进行治理，后续再通过激励与商家合作扩展平台增长路径。该闭环具备一定的合理性和可验证性，因此本项目具有较高的业务可行性。

#### 1.3.3 经济与实施可行性分析

从经济成本角度看，本文采用的主要技术方案均属于当前开发实践中常见且可控的实现路径。Vue 3、Vite、Spring Boot、MySQL 与 uni-app 均具备成熟的生态支持和较低的学习与搭建门槛，不需要额外投入高额的软件授权成本。对于毕业设计或中小型项目而言，这种技术选型能够在保证功能实现质量的同时，控制整体开发成本和时间成本。

从实施角度看，青搭的首版范围聚焦于内容消费、内容发布、活动参与、审核治理和基础激励等关键主线，没有一开始就引入支付、订单履约、复杂推荐算法或完整清结算系统等高复杂度模块，这使项目更容易在有限周期内实现较完整的可运行成果。同时，系统各端目录结构清晰，数据库脚本、初始化数据和基础部署说明已经具备，为后续联调、测试和答辩展示提供了较好的实施条件。综合来看，本课题在开发资源、时间投入与项目范围之间保持了较合理的平衡，具备良好的经济与实施可行性。

### 1.4 本文主要研究内容

本文围绕青搭校园穿搭内容平台的设计与实现展开研究，主要工作可概括为以下几个方面。

第一，对校园穿搭内容平台的需求背景进行分析，从大学生日常生活场景出发，识别当前校园穿搭内容在平台聚合、场景适配、预算友好和决策效率等方面存在的问题，并明确平台的核心用户、关键场景和产品定位。

第二，对系统需求进行梳理和结构化分析。在需求层面，本文结合场景拆解、竞品观察、原型推演、开发约束回流和测试反馈等方式，对用户端功能、后台管理能力、活动激励机制以及平台治理规则进行归纳，形成系统需求基础。

第三，对系统整体结构进行设计。本文从系统架构、业务流程、功能模块和数据库设计等方面展开，构建包括用户端、管理后台、后端服务和数据库在内的完整系统结构，并围绕内容发布、内容审核、活动参与、互动反馈和消息通知等关键链路进行流程设计。

第四，对系统进行实现与验证。本文基于 Vue 3、Spring Boot、MySQL 与 uni-app 等技术完成系统主要模块的实现，并对用户认证、内容浏览、内容发布、活动管理、后台审核、消息通知和激励相关能力进行说明，最终形成具备演示与验证价值的系统原型和工程成果。

第五，对项目实施效果进行总结，并结合当前平台结构提出后续优化方向，包括推荐机制优化、创作者成长体系完善、商家合作能力增强和数据分析能力补强等内容。

通过上述研究与实现，本文尝试构建一套兼顾用户价值、平台治理与工程落地的校园穿搭内容平台方案，为校园垂直内容产品的设计与实现提供一份较完整的实践参考。

## 第 2 章 系统需求分析

### 2.1 需求来源与目标用户分析

#### 2.1.1 需求来源分析青搭的需求来源并非单一来自功能罗列，而是在项目推进过程中逐步由场景分析、竞品观察、原型推演、开发约束和测试反馈共同形成。首先，在需求识别阶段，项目围绕大学生真实校园生活中的穿搭场景展开拆解，重点关注图书馆、自习室、早八课堂、社团活动、日常通勤等高频场景下用户对穿搭内容的参考需求。其次，通过对泛内容平台、电商平台与校园场景之间差异的观察，可以发现现有平台在真实校园语境、内容参考价值和轻决策支持方面均存在一定空缺。

在原型和功能推演阶段，项目进一步明确了平台不能仅仅停留在前台页面展示，而需要同步具备内容发布、互动反馈、活动参与和后台治理能力。随后，在工程实现过程中，又通过接口联调、数据初始化、上传限制、权限控制和消息回传等技术约束，反向补充和修正了一部分原始需求，使需求集合更加贴近真实可实现的系统边界。最后，结合测试环境体验和好友试用反馈，项目又对内容流转、审核链路、活动入口和消息通知等模块进行了进一步补充。由此可见，青搭的需求形成过程具有较明显的“场景驱动 + 闭环驱动 + 实现驱动”特点。

#### 2.1.2 目标用户分析

根据项目定位与使用场景，青搭的核心目标用户主要包括普通大学生用户、预算敏感型用户和校园内容创作者三类。

普通大学生用户是平台最广泛的基础用户群体。这部分用户通常具有日常穿搭参考需求，希望在较短时间内找到适合校园场景的搭配方案。相较于专业时尚内容，他们更关注内容是否真实、是否容易模仿、是否适合自己的校园生活状态。因此，这类用户对首页推荐内容的真实性、场景相关性和浏览效率有较高要求。

预算敏感型用户是平台中具有明显消费理性特征的一类人群。与只关注视觉表达的用户不同，这类用户更加看重单品价格、重复利用率和整体穿搭的实穿价值。他们希望平台提供的不仅是“好看”的内容，更是“可执行”的内容，即能够在有限预算内找到适合自身条件的搭配方案。因而，标签筛选、内容说明和轻导购信息对于这类用户尤为重要。

校园内容创作者则是平台内容供给的重要来源。这类用户愿意分享自己的日常穿搭、校园生活和风格表达，希望通过持续发布内容获得曝光、互动反馈和平台激励。创作者群体对发布流程、活动机制、消息通知和激励规则较为敏感，如果平台缺乏足够清晰的成长与反馈路径，创作者供给就难以持续。因此，青搭在需求设计中将内容发布、活动参与、审核回执和激励中心等能力纳入核心结构。

#### 2.1.3 平台角色分析

除普通用户外，青搭还涉及多类平台管理角色。运营人员主要负责专题活动策划、消息触达和内容扶持；审核人员负责内容审核、活动审核和异常内容治理；商家角色负责参与推广合作或专题投放；管理员则承担权限控制、平台配置、风控处理和全局运营支撑等职责。正是由于这些角色的存在，系统需求不能只关注用户端体验，还必须同步考虑后台信息架构、操作权限、状态流转和日志留痕等平台治理能力。

### 2.2 用户痛点与核心场景分析

#### 2.2.1 用户痛点分析

通过对高校场景与现有平台体验的综合分析，可以将青搭所要解决的核心痛点概括为以下几个方面。

第一，校园穿搭内容分散，缺少围绕校园日常生活的垂直聚合平台。用户虽然可以在综合内容平台上浏览大量穿搭内容，但这些内容往往缺少场景聚焦，难以直接匹配“早八怎么穿”“图书馆学习如何兼顾舒适与得体”“社团活动怎样搭配更合适”等具体问题。

第二，现有内容平台的信息参考价值存在波动。部分平台内容偏向精致展示或商业营销，虽然视觉效果较强，但与普通大学生的真实预算和使用条件存在距离。对于校园用户而言，他们更需要“今天就能照着穿”的内容，而不是脱离日常生活的理想化搭配。

第三，用户在内容消费之外还存在一定的表达需求和参与需求。许多大学生愿意记录日常穿搭、分享搭配心得或参与校园专题活动，但如果平台缺少低门槛发布机制、明确的活动入口和及时反馈机制，用户表达意愿就难以转化为稳定的内容供给。

第四，平台若缺少审核与治理能力，内容质量将难以保障。校园内容平台具有明显的社区属性，一旦内容质量失控、重复灌水增多或激励机制被滥用，用户体验就会迅速下降。因此，在需求分析阶段就必须把审核、活动规则和异常处理能力纳入系统整体考虑。

第五，平台若缺少活动和激励支撑，冷启动阶段很容易陷入“没有内容就没有用户，没有用户就没有内容”的困境。青搭并非单纯浏览型产品，而是需要通过活动参与和激励反馈推动创作者持续产出。因此，需求分析不能停留在内容浏览层面，而要将增长和治理逻辑一并纳入系统设计。

#### 2.2.2 核心使用场景分析青搭的核心使用场景主要围绕大学生的高频校园生活展开。第一类场景是课堂与学习场景，包括早八上课、图书馆、自习室等，用户在这些情境下更关注舒适、简洁、得体和低门槛复制的穿搭方案。第二类场景是日常通勤场景，即从宿舍到教学楼、食堂或实验楼的常规校园流动，这一场景强调穿搭的实用性和季节适配性。第三类场景是社团活动、校园聚会、拍照打卡等相对具备表达属性的场景，用户在此类情境下更重视个性展示和风格识别。第四类场景是平台创作与参与场景，即用户浏览内容后产生互动、模仿或发布行为，并进一步参与专题活动、关注创作者或接收消息通知。

这些场景的共同特点在于高频、真实、容易共鸣，同时也较适合平台通过标签、专题和活动进行结构化组织。正因如此，青搭在需求上不仅需要支持内容浏览，还需要支持标签筛选、活动绑定、消息通知和个人中心沉淀，使用户能够围绕不同校园场景持续消费和生产内容。

### 2.3 功能需求分析

#### 2.3.1 用户端功能需求

用户端功能是青搭的核心使用入口，其设计目标是帮助用户高效完成内容浏览、互动参与和内容发布三类关键行为。

在基础能力方面，系统需要提供登录、注册和资料完善功能，用于建立用户身份、记录基础信息并支撑后续互动与发布操作。对于未登录用户，可适度开放内容浏览能力，但点赞、评论、收藏、关注、发布和参与激励等关键行为应要求用户先完成身份确认。

在内容消费方面，系统需要提供首页内容流、内容详情页、搜索和标签筛选能力。首页内容流应支持按校园场景、标签或活动专题展示内容卡片，使用户能够快速进入浏览状态；详情页应承担图文展示、标签说明、互动操作和活动信息承接等作用；搜索与筛选功能则用于降低内容查找成本，使用户能够按照场景、预算、风格或主题快速定位目标内容。

在互动与沉淀方面，系统需要支持点赞、评论、收藏、关注和消息通知等功能。互动功能用于提升用户停留时长与内容参与度，消息通知则承担审核结果、互动反馈、活动提醒等回传职责，保证用户行为能够形成持续反馈。与此同时，个人主页、我的发布、我的收藏、我的活动和资料设置等能力，能够帮助用户沉淀个人内容资产和互动记录。

在内容创作方面，系统需要支持发布内容、上传图片、填写标题与正文、选择标签或专题、保存草稿和提交审核等功能。其中，草稿机制对于降低创作中断损失和提升发布成功率具有实际意义；审核机制则是保证平台内容质量、活动合规性和激励公平性的前提条件。因此，用户端功能不仅要满足浏览需求，也要兼顾内容生产和平台治理所需的状态流转。

#### 2.3.2 管理后台功能需求

管理后台是青搭平台治理能力的核心承载端，其目标是为运营、审核和管理员角色提供统一、清晰且可执行的管理入口。

首先，后台需要具备管理员登录和权限控制能力，以区分不同角色的访问范围和操作边界。其次，系统应提供数据看板，用于查看内容数量、用户规模、活动表现和结算相关的核心指标，为平台运营与复盘提供数据支撑。

在内容治理方面，后台需要具备内容审核能力，包括查看待审内容、执行通过或驳回操作、记录审核状态和返回处理结果。对于活动和公告模块，后台还应提供活动创建与编辑、专题配置、官方公告发布和消息触达等功能，以支撑平台日常运营。对于商家和合作模块，后台需要支持商家信息管理、合作记录维护和推广资源管理，以承接后续轻商业化需求。

在平台经营方面，后台还需要提供结算记录查看、提现申请审核和必要的异常处理能力。虽然首版系统并未引入完整的支付与清结算体系，但通过结算记录和提现审核等模块，可以初步体现平台在创作者激励和经营管理方面的结构能力，使整体系统更加完整。

#### 2.3.3 平台运营与激励功能需求

与一般内容浏览型应用不同，青搭将活动和激励机制视为平台的重要增长支撑，因此这一部分需求需要单独分析。

首先，系统需要提供活动中心或专题入口，使用户能够了解当前活动主题、参与条件和奖励逻辑。其次，发布内容时应支持活动绑定或专题选择，以便平台将用户内容纳入活动统计范围。再次，系统需要具备对活动内容进行审核、统计和状态追踪的能力，确保进入激励结算范围的内容具备可追溯性。

此外，平台还需要提供消息反馈和激励中心能力。消息反馈用于告知用户审核结果、活动提醒和互动通知，激励中心则用于呈现收益记录、激励说明、提现申请等与创作者回报相关的信息。通过这套机制，平台可以逐步形成“内容发布 - 审核通过 - 参与活动 - 获得反馈 - 持续创作”的闭环，这也是青搭与传统纯展示型原型项目的重要区别。

### 2.4 非功能需求分析

#### 2.4.1 易用性需求青搭面向的主要用户为大学生群体，因此系统在交互上应尽量保持清晰、轻量和低学习成本。首页应帮助用户快速识别内容主题与场景标签，发布流程应尽量减少输入阻力，后台操作则应突出列表、筛选和状态操作的直观性。良好的易用性有助于提高内容浏览效率、降低首篇发布门槛，并减少管理侧使用成本。

#### 2.4.2 可维护性需求

由于系统同时包含用户端、管理后台、后端服务和数据库脚本，项目在工程结构上需要保持模块清晰、职责分离和接口边界明确。前后端分离架构、独立目录管理和统一 API 调用方式，有助于降低系统后续修改难度。数据库脚本和初始化数据的独立维护，也能够提升系统在联调、重建环境和后续扩展时的便利性。

#### 2.4.3 可扩展性需求青搭的首版目标是验证内容平台闭环，因此当前未引入复杂推荐算法、完整交易系统和高级成长体系。但从长期演进来看，系统应具备继续增加推荐能力、榜单能力、商家投放能力和更细粒度数据分析能力的可能。因此，在需求设计上需要保证标签体系、活动结构、消息机制和后台配置能力具备一定的扩展空间。

#### 2.4.4 安全性需求

系统涉及用户身份、内容管理、审核操作和后台权限，因此需要保证基础安全性。前端与后端之间的接口访问需要具备基本身份校验机制，后台操作需要通过管理员权限隔离来控制访问范围，上传和数据写入过程也需要进行必要的校验和异常处理。对于活动激励和结算相关能力，还应保留异常行为识别与人工复核的处理空间，以避免平台规则被滥用。

#### 2.4.5 性能与稳定性需求

作为一个内容平台，青搭在性能层面的关键诉求主要集中于首页内容加载、详情页打开速度、发布提交响应和后台列表查询效率。虽然本项目规模尚不需要采用复杂的高并发架构，但系统仍应保证在演示和测试环境下具备稳定可运行能力。通过合理的数据结构设计、接口拆分、前端异步请求和基础异常处理，可以在现有项目规模下满足论文答辩与系统展示所需的性能与稳定性要求。

综上所述，青搭的需求并不是围绕单一页面展开，而是围绕“内容、互动、活动、审核、激励和治理”六类能力共同构成。只有当用户端体验、后台管理能力和平台规则支撑同时具备时，系统才能形成一个结构完整、可运行、可验证的校园穿搭内容平台。
